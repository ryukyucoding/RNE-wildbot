# WildBot 實作方法介紹

## 一、物件抓取任務實作方法介紹

物件抓取任務（Bear Mission）是整個 WildBot 系統中最複雜的自動化流程，由 `bear_mission_node.py`（3,200+ 行）實現完整的「記錄起點 → 搜尋 → 逼近 → 夾取 → 返航 → 放下」閉環任務。系統支援 `repeat_mission` 模式，能在完成一輪後立即開始下一輪，直到手動中止。

### 1.1 系統架構與感測器融合

系統運行在多 Docker 容器的微服務架構上，各容器透過 ROS 2 DDS 通訊：
- **YOLO 容器**（`ros2_yolo_integration`）：執行物件偵測，發佈 `/yolo/target_info`（偵測狀態、深度、像素偏移、bbox 寬高）。
- **車控容器**（`pros_car`）：執行任務狀態機與底盤控制。
- **定位容器**（`pros_app`）：SLAM / AMCL 定位與 Nav2 導航堆疊。

感測器融合方面，我們實作了 **ObstacleGuard**（`obstacle_guard.py`，800+ 行），將 LiDAR 三扇區（前方、左側、右側）與深度相機五扇區（前、前左、左、前右、右）進行即時融合。融合邏輯能智慧判斷「LiDAR 讀到的是地面還是牆壁」、「深度相機的低值是否為地板假陽性」等邊緣情況，在逼近模式下會動態調整判斷邏輯，避免將目標物本身誤判為障礙。

### 1.2 YOLO 物件偵測與後處理

我們使用自訓練的 YOLOv11 模型（`detection.pt`），能辨識 `bear`（黑熊玩偶）與 `knob`（門把）兩個類別。

**抗雜訊後處理**：為解決場地邊緣招牌上粗黑字體被誤判為目標的問題，我們在推論後處理階段加入了**長寬比過濾（Aspect Ratio Filter）**。由於招牌文字的 Bounding Box 多呈細長條狀（如 w/h > 5），而熊玩偶較方正（通常 0.5 ~ 2.0），系統預設只保留長寬比介於 `0.3 ~ 3.0` 之間的偵測結果。此過濾條件透過 ROS Parameters（`bbox_min_aspect_ratio`、`bbox_max_aspect_ratio`）動態調整，不需重新編譯即可在現場微調。被過濾的框會在偵錯影像上以紅色（長寬比不符）標示，方便現場確認。

### 1.3 視覺伺服逼近（Visual Servoing）

鎖定目標後，系統進入連續式 PID 視覺伺服控制，直接輸出左右輪速（取代傳統的 bang-bang 離散指令）：

- **轉向控制**：以像素偏差 `dx_px` 透過 `compute_yaw_wheel_from_pixel()` 函式計算轉向輪速。此函式使用分段 `tanh` 曲線實現非線性映射——小偏差時給予柔和轉向避免 overshoot，大偏差時保證最小轉向力避免原地卡住。同時內建 overshoot 阻尼（當 dx 變化率與偏差方向相反時自動減速）。
- **前進控制**：採用 PID 控制器（Kp=200, Ki=8, Kd=8）計算前進速度，並依距離分三個速度區段：
  - **遠距（>1.5m）**：最高速 390 wheel units 全速接近
  - **中距（0.8~1.5m）**：降至 170 wheel units 精準轉向
  - **近距（<0.4m）**：進入夾取準備區
- **EMA 濾波**：對像素偏差與深度值都套用指數移動平均（Exponential Moving Average）濾波，分別以 `dx_ema_alpha=0.25` 與 `depth_ema_alpha=0.35` 的係數平滑 YOLO 幀間跳動。

### 1.4 多重夾取觸發條件

為確保在各種感測器失效情況下都能可靠夾取，系統設計了五種互補的觸發機制：

1. **連續幀確認**：深度持續在 grasp zone 內超過 4 幀才觸發（過濾深度突波）。
2. **深度失效觸發**：偵測到目標但深度回報 -1（太近，< 40cm，感測器失效）→ 直接夾取。
3. **Bbox 面積觸發**：bbox 寬度超過 200px → 視為已足夠近（深度不可靠時的備用判斷）。
4. **消失記憶觸發**：目標在近距離消失（被夾爪擋住視線），最後已知距離 ≤ 0.56m → 根據記憶位置執行夾取。
5. **深度跳遠觸發**：深度突然從近處跳到遠處（YOLO 換成下一隻熊或太近重選）→ 立即煞停夾取。

### 1.5 機械臂控制（2D IK）

手臂控制器（`arm_controller_2D.py`）管理三軸手臂（Shoulder、Elbow、Gripper），具備：
- **2D 逆向運動學（IK）**：以兩連桿模型（L1=8.09cm, L2=11cm）計算 Shoulder 與 Elbow 的目標角度，採用 Elbow-Up 構型（由上往下夾）。
- **平滑軌跡插值（Trajectory Interpolation）**：每次最多移動 2~5°，搭配 50ms~100ms 延遲，實現絲滑的關節運動，避免伺服馬達過載。
- **夾取序列**：Elbow 微降 12° → 夾爪合緊 → Shoulder 上抬 25° 固定。
- **釋放序列**：開夾爪讓熊落地 → 等 0.8 秒穩定 → 手臂收回安全姿態。

### 1.6 多策略返航機制

夾取完成後，系統會嘗試多種方式返回起點：

1. **Nav2 NavigateToPose**（首選）：使用記錄的 AMCL home pose 發送導航目標。回程中持續監控障礙物，若前方或側面被完全擠壓，會主動取消當前目標。
2. **Nav2 卡住重試**：若 AMCL 距離 home 長時間（8 秒）無進展，主動取消 → 清除 costmap → 執行脫困機動（後退 0.8 秒 + 旋轉 0.8 秒）→ 重新導航。
3. **Fallback 直駛回家**：當 Nav2 的 bt_navigator 未啟動時，改用純 AMCL pose 計算朝向 home 的角度差，以「旋轉對準 → 前進一段 → 再旋轉」的方式逐步逼近。
4. **Odom 回程**（末選）：完全沒有 AMCL 時，使用里程計推估回程路徑。

### 1.7 YOLO 丟失搜尋狀態機

當目標在逼近過程中丟失，系統啟動多層搜尋策略：

- **Grace Period（1.5 秒）**：短暫丟失時沿用上一幀速度繼續前進，不急煞。
- **近距離後退重找（Back Peek）**：若最後已知距離在夾取範圍附近，先後退 0.55 秒嘗試重新捕捉（最多 2 次）。
- **前進 + 右轉搜尋循環**：前進 3 秒 → 右轉 30° → 前進 3 秒 → 右轉 30°，持續循環直到重新捕捉。
- **通用脫困機動**：若 AMCL 位移長時間無進展，執行「後退 → 側移 → 前進」的 S 型脫困。

## 二、上橋取物任務實作方法介紹

上橋任務（Bridge Navigation）在 `feat/bridge-navigation` 分支獨立開發，結合 **SLAM 持續定位** 與 **自訂 ROS 2 Action Server 分段式過橋狀態機**，以應對橋面斜坡打滑與 LiDAR 特徵消失的問題。

### 2.1 環境建圖與參數標定

任務開始前需進行一次性的場景建圖與座標標定：

1. **SLAM 建圖**：啟動 SLAM 容器，手動控制車子在場地繞行一圈，在 Foxglove 中確認 `/map` 已將橋與周圍牆壁掃描清楚。
2. **橋梁座標量測**：在 Foxglove 3D Panel 中使用 Publish Point 工具，分別點擊橋頭（Point A，橋入口前平地中線）與橋尾（Point B，橋出口後平地中線），透過 `ros2 topic echo /clicked_point` 讀取座標值。
3. **中軸線角度計算**：以 `atan2(By - Ay, Bx - Ax)` 計算橋梁中軸線航向角（`bridge_heading_deg`），供後續對齊使用。
4. **參數填入**：將以上數值填入 `bridge_params.yaml`，包含橋頭座標 (x, y)、航向角、到達門檻 (`foot_reached_thresh_m: 0.3`)、對齊容差 (`align_tol_deg: 8.0`)、以及各段過橋時間與動作指令。

### 2.2 過橋狀態機（FSM）

任務由 `car_nav_controller.py` 中的 `bridge_nav()` 方法實作，包含四個狀態：

#### APPROACH（導航至橋頭）
- 將橋頭座標與航向角發佈至 Nav2 的 `/goal_pose`。
- 等待 Nav2 將車子導航至橋頭附近（距離 < `foot_reached_thresh_m`）。

#### ALIGN（原地旋轉對齊）
- 透過 TF2 讀取當前車身 yaw 角，計算與橋梁中軸線的角度差（`diff`）。
- 若 `|diff| < align_tol_deg`（8°），視為對齊完成。
- 旋轉速度自適應：角度差 > 30° 用高速旋轉，< 30° 切換中速旋轉，確保精準收斂。

#### CROSS（分段式過橋）
以時間為基準的三段式控制，每一段獨立設定動作與持續時間：

| 階段 | 動作指令 | 預設時間 | 設計考量 |
|------|----------|----------|----------|
| **CROSS_UP**（上坡） | `FORWARD`（全速） | 3.0 秒 | 需要足夠動力克服重力與摩擦力 |
| **CROSS_PLATFORM**（平臺） | `FORWARD_SLOW`（慢速） | 2.0 秒 | 穩定行駛，準備夾取橋上物件 |
| **CROSS_DOWN**（下坡） | `FORWARD_SLOW`（慢速） | 3.0 秒 | 防止下坡暴衝翻車 |

#### DONE（完成）
- 連續發送 5 次 STOP 確保完全煞停。
- 清除導航計畫與目標位姿。

### 2.3 持續 SLAM 定位策略

整個過橋任務期間，**SLAM 容器從頭到尾保持運行**，不切換至純 AMCL 模式。原因是橋上的 LiDAR 掃描經常因為斜坡角度或缺乏特徵物而導致 AMCL 粒子發散。SLAM 的持續運行能利用里程計與增量式地圖更新，提供更穩定的定位。落地後等待數秒讓 SLAM 重新收斂，即可恢復正常精度。

## 三、開門任務實作方法介紹

開門任務（Door Open Task）在 `open_door` 分支獨立開發，由 `door_open_task.py` 實現完整的 8 狀態有限狀態機（FSM），從導航至門前、偵測門把、視覺伺服對齊、到機械臂下壓開門，全程自動化。

### 3.1 狀態機流程總覽

```
State 0: NAVIGATE_TO_DOOR  → Nav2 導航至門前預設座標
State 1: SEARCH_HANDLE     → 原地旋轉搜尋門把（最多 60 秒）
State 2: VISUAL_SERVO      → PID 視覺伺服逼近 + 精對齊
State 4: ARM_AIM           → 手臂移至門把正上方
State 5: PRESS_DOWN        → 夾爪合起 → Z 軸下壓 7cm
State 6: OPEN_DOOR         → 車體前進/後退推開門
State 7: DONE              → 重置手臂，任務完成
```

### 3.2 門把辨識

使用與物件抓取相同的 YOLO 模型（`detection.pt`），透過 `/target_label` topic 通知 YOLO 節點將追蹤目標切換為 `knob`（門把）。同樣可利用長寬比過濾排除環境文字干擾。

### 3.3 視覺伺服逼近（Visual Servo Approach）

本狀態將舊版的「先對齊再前進」兩步驟合併為**單一閉迴路**：同時計算前進與轉向，讓車子在靠近的過程中持續保持門把在畫面中央。

- **逼近階段**：PID 控制器根據像素偏差計算左右輪速差（轉向），同時以基礎前進速度（60 wheel units）推進。最大轉速差限制 250，最小保底 120（克服靜摩擦力死區）。
- **深度 EMA 平滑**：以 `alpha=0.3` 平滑 YOLO 深度值，避免單幀突波導致過早停車。
- **停車條件**：LiDAR 前方最小距離 ≤ 0.30m **或** YOLO EMA 深度 ≤ 0.30m。
- **YOLO 丟失耐心**：門把短暫丟失時維持當前速度 3 秒，超時則退回搜尋狀態。

### 3.4 精對齊（Fine Alignment）

停車後進入純旋轉微調階段：
- 只做原地旋轉（前進速度為 0），以 `VS_ALIGN_KP=0.6` 的比例係數修正像素偏差。
- 當 `|error| < 20px` 時視為對齊完成。
- 精對齊最長等待 5 秒，超時則強制進入手臂動作（避免因微小抖動無限循環）。

### 3.5 手臂定位與下壓

#### ARM_AIM（State 4）
利用已知的停車距離與結構參數直接計算手臂目標位置，不依賴即時 YOLO：
- 手臂目標 X 軸距離 = 停車深度（0.30m）+ 攝影機到手臂基座的結構補償（`CAMERA_X_OFFSET = -0.15m`）= **0.15m**
- 此距離完美落在手臂最大可達距離（`ARM_MAX_REACH = 0.17m`）內。
- 先收縮至安全姿態（避免卡門把），再伸出至門把正上方 5cm（`PRESS_ABOVE_OFFSET = 0.05m`）。

#### PRESS_DOWN（State 5）
- 夾爪合起至接近極限（10° → 退回 12° 避免燒壞馬達）。
- 手臂在 Z 軸直接下壓 7cm（`PRESS_Z_DOWN = 0.07m`，涵蓋門把行程 5~8cm）。

#### OPEN_DOOR（State 6）
- 維持下壓姿態，車體低速前進（或後退，依門開向設定 `DOOR_OPEN_ACTION`）持續 2.5 秒，利用車體推力將門順利推開。

### 3.6 任務結束與安全重置

無論任務成功或失敗，FSM 都會：
1. 清除 YOLO 追蹤目標（`publish_target_label("")`）。
2. 釋放夾爪（張開至 70°）。
3. 重置手臂至初始姿態（全部回歸 90°）。

## 四、硬體改裝介紹 (optional)

無硬體改裝。本團隊採用標準 Wildbot 設備與感測器。

我們透過精準的軟體參數標定——包含攝影機到手臂基座的 X 軸偏移（`CAMERA_X_OFFSET`）、門把相對手臂基座的固定高度（`KNOB_Z_HEIGHT`）、以及各關節的角度補償（`offset`）與方向乘數（`dir`）——在不改裝硬體的前提下，實現高精度的末端定位。

## 五、其他特殊設計 (optional)

### 5.1 Docker 模組化微服務架構

我們將系統拆分為三個獨立 Docker 容器：

| 容器 | 職責 | 主要 Package |
|------|------|-------------|
| `pros_app` | 核心通訊、SLAM、Nav2、rosbridge | slam_unity, rosbridge_server |
| `pros_car` | 任務狀態機、車控、手臂控制 | pros_car_py, car_control_pkg |
| `ros2_yolo_integration` | YOLO 視覺辨識、深度估計 | yolo_example_pkg |

各容器透過共用的 Docker bridge network 進行 ROS 2 DDS 通訊，避免 Python 套件與 CUDA 版本衝突，同時允許團隊成員平行開發與獨立測試。

### 5.2 Digital Twin 虛實整合開發

在部署至實體機器人前，我們大量使用 `pros_twin`（Unity）進行數位孿生模擬。Unity 場景模擬了完整的競賽場地（含橋梁、門把、黑熊玩偶），車子透過 rosbridge WebSocket 與 Docker 內的 ROS 2 系統通訊。這讓我們能在 macOS 環境安全地測試：

- 導航路徑規劃與避障邏輯
- 視覺伺服的 PID 參數調校
- 手臂 IK 抓取序列的驗證
- 過橋打滑與下坡暴衝的極端場景

大幅減少實車測試的風險與時間成本。

### 5.3 動態參數調整（Dynamic Parameters）

所有關鍵行為參數——PID 增益、速度閾值、距離門檻、YOLO 過濾條件、障礙護欄距離——皆透過 ROS 2 Parameters 宣告，支援：
- **啟動時覆寫**：`ros2 run ... --ros-args -p visual_servo_max_forward_speed:=200.0`
- **YAML 檔案載入**：如 `bridge_params.yaml` 集中管理過橋參數
- **環境變數注入**：如 `YOLO_CONF=0.3` 調整偵測信心門檻

這讓我們能在不重新編譯的情況下，依據實車測試的當下環境與光線條件即時調整系統行為。

### 5.4 ObstacleGuard 多源感測器安全護欄

`ObstacleGuard`（800+ 行）提供了從「未知環境安全前進」到「逼近目標時不誤擋」的完整防護：

- **LiDAR + 深度相機融合**：分前方、前左、左、前右、右、後方共 6 區段，取最近有效值。
- **車身反射過濾**：在逼近模式下，自動辨識深度相機側面讀到的低值是否為車身後輪反射，而非真實障礙。
- **窄通道減速**：根據左右間距自動計算走廊縮放係數（corridor_scale），在狹窄處自動降速。
- **逃脫方向決策**：前方受阻時，自動選擇左右較開闊的方向旋轉脫離。
- **後方安全**：後退時同樣受後方 LiDAR / 深度保護，防止後撞。
