# WildBot 實作方法介紹

## 一、物件抓取任務實作方法介紹

我們的物件抓取任務（Bear Mission）採用 **YOLO26 物件偵測**結合 **ROS 2 Nav2 導航與視覺伺服（Visual Servoing）** 技術來完成自動化夾取與放置。

1. **目標辨識**：系統透過整合在 Docker 容器中的 YOLO 節點訂閱相機影像，並載入訓練好的模型（`detection.pt`）來即時辨識目標物（Bear）。辨識結果會輸出目標的像素偏移與深度資訊（`/yolo/target_info`）。
2. **自動導航與避障**：啟動任務時，機器人會先記錄初始的 Home 座標。接著系統利用 AMCL 進行定位，並透過 Nav2 產生全域路徑與局部避障（Obstacle Guard）。
3. **視覺伺服逼近**：當機器人接近目標且目標進入視野後，進入視覺伺服模式。系統根據 YOLO 提供的水平像素誤差與深度，即時調整車體旋轉與前進速度，使車身精準對齊目標。
4. **夾取與返航**：當深度達到夾取觸發距離（如 0.5m）內，狀態機發送夾取指令給機械臂。夾取成功後，自動透過 Nav2 導航回起點（Home）並放下物件。

## 二、上橋取物任務實作方法介紹

上橋任務結合了 **SLAM 即時定位**與**自訂狀態機時間控制**，以應對橋面斜坡打滑與視線死角的問題。

1. **環境建圖與參數設定**：首先進行場景 SLAM 建圖，並在 Foxglove 中量測橋頭（Point A）與橋尾（Point B）的座標，以此計算出橋梁的中軸線角度（Heading）。這些參數統一存於 `bridge_params.yaml` 中。
2. **精準導航與對齊**：任務啟動後，使用 Nav2 導航至橋頭的精確座標。到達橋頭後，機器人會進行原地旋轉，將車身與橋梁中軸線完美對齊（角度誤差容許值 `align_tol_deg` 設為 8 度內）。
3. **分段式過橋狀態機**：為了克服上坡打滑與下坡暴衝，我們實作了自訂的 `Bridge_Nav` Action Server 進行分段控制：
   - **上坡段（CROSS_UP）**：設定全速前進（FORWARD）並維持特定秒數，確保有足夠動力爬升。
   - **平臺段（CROSS_PLATFORM）**：轉為慢速前進（FORWARD_SLOW）。
   - **下坡段（CROSS_DOWN）**：以慢速穩定下坡，確保安全著陸。
4. **持續定位**：過橋期間不切換至純 AMCL，而是持續運行 SLAM 以避免上坡時 LiDAR 掃不到特徵物造成的定位丟失。

## 三、開門任務實作方法介紹

開門任務採用了與物件抓取相似的視覺對齊框架，並針對門把特徵進行了最佳化。

1. **門把辨識**：使用專為門把（Knob）訓練的 YOLO 模型，將影像轉換為門把的三維空間相對位置。
2. **初步定位與搜尋**：機器人首先會手動或透過導航移動至門前的預設點（`DOOR_APPROACH_GOAL`）。若視野內未發現門把，系統設計了搜尋機制（最大等待與旋轉 `SEARCH_MAX_ITER`）來尋找門把。
3. **精確對齊**：一旦鎖定門把，系統以視覺伺服技術將車體中心點與門把對齊（容差值設為 `ALIGN_PIXEL_TOL`）。
4. **開門操作**：車體與門把對準且距離達到設定閾值後，觸發開門狀態機，機械臂會依照預設的軌跡伸出並轉動門把，最後配合車體後退或移動拉開/推開房門。

## 四、硬體改裝介紹(optional)

無硬體改裝。本團隊採用大會提供的標準 Wildbot 設備與感測器。

## 五、其他特殊設計(optional)

1. **Docker 模組化微服務架構**：
   我們將系統拆分為 `pros_app`（核心通訊與 SLAM）、`pros_car`（控制狀態機）與 `ros2_yolo_integration`（視覺辨識）等多個獨立 Docker 容器。此設計避免了套件衝突，並允許團隊成員平行開發與測試。
2. **Digital Twin 虛實整合開發**：
   在部署至實體機器人前，我們大量使用了 `pros_twin` (Unity) 進行數位孿生模擬。這讓我們能夠在 macOS 本機環境安全地測試各種導航邏輯、視覺伺服與極端狀態（如橋上打滑），大幅減少實車測試的風險與時間成本。
